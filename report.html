Project: Train a Quadcopter How to Fly¶

Design an agent to fly a quadcopter, and then train it using a reinforcement learning algorithm of your choice!

Try to apply the techniques you have learnt, but also feel free to come up with innovative ideas and test them.

Instructions¶

Take a look at the files in the directory to better understand the structure of the project.

  • agents/: Folder containing reinforcement learning agents.

You are welcome to create any additional files to help you to organize your code. For instance, you may find it useful to define a model.py file defining any needed neural network architectures.

Controlling the Quadcopter¶

We provide a sample agent in the code cell below to show you how to use the sim to control the quadcopter. This agent is even simpler than the sample agent that you'll examine (in agents/policy_search.py) later in this notebook!

The agent controls the quadcopter by setting the revolutions per second on each of its four rotors. The provided agent in the Basic_Agent class below always selects a random action for each of the four rotors. These four speeds are returned by the act method as a list of four floating-point numbers.

For this project, the agent that you will implement in agents/agent.py will have a far more intelligent method for selecting actions!

import random

class Basic_Agent():
    def __init__(self, task):
        self.task = task
    
    def act(self):
        new_thrust = random.gauss(450., 25.)
        return [new_thrust + random.gauss(0., 1.) for x in range(4)]

import csv
import numpy as np
from task import Task

# Modify the values below to give the quadcopter a different starting position.
runtime = 5.                                     # time limit of the episode
init_pose = np.array([0., 0., 10., 0., 0., 0.])  # initial pose
init_velocities = np.array([0., 0., 0.])         # initial velocities
init_angle_velocities = np.array([0., 0., 0.])   # initial angle velocities
file_output = 'data.txt'                         # file name for saved results

# Setup
task = Task(init_pose, init_velocities, init_angle_velocities, runtime)
agent = Basic_Agent(task)
done = False
labels = ['time', 'x', 'y', 'z', 'phi', 'theta', 'psi', 'x_velocity',
          'y_velocity', 'z_velocity', 'phi_velocity', 'theta_velocity',
          'psi_velocity', 'rotor_speed1', 'rotor_speed2', 'rotor_speed3', 'rotor_speed4']
results = {x : [] for x in labels}

# Run the simulation, and save the results.
with open(file_output, 'w') as csvfile:
    writer = csv.writer(csvfile)
    writer.writerow(labels)
    while True:
        rotor_speeds = agent.act()
        _, _, done = task.step(rotor_speeds)
        to_write = [task.sim.time] + list(task.sim.pose) + list(task.sim.v) + list(task.sim.angular_v) + list(rotor_speeds)
        for ii in range(len(labels)):
            results[labels[ii]].append(to_write[ii])
        writer.writerow(to_write)
        if done:
            break

The Task¶

A sample task has been provided for you in task.py. Open this file in a new window now.

The __init__() method is used to initialize several variables that are needed to specify the task.

  • The simulator is initialized as an instance of the PhysicsSim class (from physics_sim.py).
  • Inspired by the methodology in the original DDPG paper, we make use of action repeats. For each timestep of the agent, we step the simulation action_repeats timesteps. If you are not familiar with action repeats, please read the Results section in the DDPG paper.
  • We set the number of elements in the state vector. For the sample task, we only work with the 6-dimensional pose information. To set the size of the state (state_size), we must take action repeats into account.
  • The environment will always have a 4-dimensional action space, with one entry for each rotor (action_size=4). You can set the minimum (action_low) and maximum (action_high) values of each entry here.
  • The sample task in this provided file is for the agent to reach a target position. We specify that target position as a variable.

The reset() method resets the simulator. The agent should call this method every time the episode ends. You can see an example of this in the code cell below.

The step() method is perhaps the most important. It accepts the agent's choice of action rotor_speeds, which is used to prepare the next state to pass on to the agent. Then, the reward is computed from get_reward(). The episode is considered done if the time limit has been exceeded, or the quadcopter has travelled outside of the bounds of the simulation.

In the next section, you will learn how to test the performance of an agent on this task.

The Agent¶

The sample agent given in agents/policy_search.py uses a very simplistic linear policy to directly compute the action vector as a dot product of the state vector and a matrix of weights. Then, it randomly perturbs the parameters by adding some Gaussian noise, to produce a different policy. Based on the average reward obtained in each episode (score), it keeps track of the best set of parameters found so far, how the score is changing, and accordingly tweaks a scaling factor to widen or tighten the noise.

Define the Task, Design the Agent, and Train Your Agent!¶

Amend task.py to specify a task of your choosing. If you're unsure what kind of task to specify, you may like to teach your quadcopter to takeoff, hover in place, land softly, or reach a target pose.

After specifying your task, use the sample agent in agents/policy_search.py as a template to define your own agent in agents/agent.py. You can borrow whatever you need from the sample agent, including ideas on how you might modularize your code (using helper methods like act(), learn(), reset_episode(), etc.).

Note that it is highly unlikely that the first agent and task that you specify will learn well. You will likely have to tweak various hyperparameters and the reward function for your task until you arrive at reasonably good behavior.

As you develop your agent, it's important to keep an eye on how it's performing. Use the code above as inspiration to build in a mechanism to log/save the total rewards obtained in each episode to file. If the episode rewards are gradually increasing, this is an indication that your agent is learning.

import sys
import pandas as pd
from agents.agent import DDPG
from task import Task

num_episodes = 400
target_pos = np.array([0., 0., 470.])
task = Task(init_pose=np.array([0.,0.,10.,0.,0.,0.]),target_pos=target_pos)

agent = DDPG(task) 
file_output = 'data.txt' 
labels = ['time', 'x', 'y', 'z', 'phi', 'theta', 'psi', 'x_velocity',
          'y_velocity', 'z_velocity', 'phi_velocity', 'theta_velocity',
          'psi_velocity', 'rotor_speed1', 'rotor_speed2', 'rotor_speed3', 'rotor_speed4','reward']
results = [{x : [] for x in labels} for i in range (num_episodes)]

for i_episode in range(1, num_episodes+1):
    state = agent.reset_episode() # start a new episode
    
    while True:
        action = agent.act(state)
        next_state, reward, done = task.step(action)
        agent.step(action,reward,next_state,done)
        state = next_state
        for ii in range(len(labels)):
            to_write = [task.sim.time] + list(task.sim.pose) + list(task.sim.v) + list(task.sim.angular_v) + list(action)+[reward]
            results[i_episode-1][labels[ii]].append(to_write[ii])
        if done:
            x=agent.actor_target.model.predict(np.array([0.]).astype(np.float32).reshape(-1,1))[0]
            q=agent.critic_target.model.predict([np.array([0.]).astype(np.float32).reshape(-1,1),np.array(x).reshape(-1,4)])[0]
            print("\rEpisode = {:4d}, score = {:7.3f} (best = {:7.3f}) (height={:7.2f}) (actions={:7.2f}{:7.2f}{:7.2f}{:7.2f}) (Q(s,a)=  {:7.2f})".format(
                i_episode, agent.score, agent.best_score,task.sim.pose[2],x[0],x[1],x[2],x[3],q[0]), end="")  # [debug]
            if agent.best_score==agent.score:
                print()
            break
            
    sys.stdout.flush()

Reflections¶

Question 1: Describe the task that you specified in task.py. How did you design the reward function?

Answer:

The quadcopter just won't fly in the initial 2-3 days. So the task was to make it fly as high as possible :). Lot of different reward functions were tried, at the end this simple reward function seems to have done the trick-

reward=
max(-0.2,min(0.1,(self.sim.v[2]-vel_prev[2])/100))
+max(-0.2,min(self.sim.v[2]/100,0.5))
-min(0.3,(abs(self.sim.angular_v[0]-ang_vel_prev[0])+abs(self.sim.angular_v[1]-ang_vel_prev[1])+abs(self.sim.angular_v[2]-ang_vel_prev[2]))/100)

Basically higher the acceleration (self.sim.v[2]-vel_prev[2] should give the acceleration as difference is taken b/w successive steps of the task so time difference is constant) and velocity (self.sim.v[2]) in the z direction, higher will be the reward. The quadcopter is also penalized if it has angular acceleration in any direction.

Question 2: Discuss your agent briefly, using the following questions as a guide:

  • What learning algorithm(s) did you try? What worked best for you?
  • What was your final choice of hyperparameters (such as $\alpha$, $\gamma$, $\epsilon$, etc.)?
  • What neural network architecture did you use (if any)? Specify layers, sizes, activation functions, etc.

Answer:

I mainly tried the sample DDPG agent that was provided with the project. A lot of time went in trying to understand the sample code and why it was written that way. The DDPG paper that was linked with the project helped. But still the loss calculation for Actor model is not very clear to me (why is there a -ve sign in "loss = K.mean(-action_gradients * actions)"). I did add batch normalization layers to Actor model as from what I read, it should help in converging the model quickly.

Initially agent would crash within a second or two so I thought maybe it was not able to learn as it only had bad experiences in the replay buffer. So I tried prioritized replay but that didn't help much either.

Main problem faced by me was that I wasn't sure if the critic and actor model were learning properly or not. So to have some idea regarding the Q value that the critic model should converge to, I tried having $\gamma$=0 so that the q value will be equivalent to reward and because reward function was not too complicated so optimum reward value could be easily calculated. But this also didn't help as the agent was not converging much and every few episodes there will be a new behavior.

At the end, these were the hyperparameter values that worked-

    #noise parameters
    self.exploration_mu = 0
    self.exploration_theta = 0.15*5
    self.exploration_sigma = 0.2*5

    # Algorithm parameters
    self.gamma = 0.9  # discount factor
    self.tau = 0.2  # for soft update of target parameters

Question 3: Using the episode rewards plot, discuss how the agent learned over time.

  • Was it an easy task to learn or hard?
  • Was there a gradual learning curve, or an aha moment?
  • How good was the final performance of the agent? (e.g. mean rewards over the last 10 episodes)

Answer:

Once things (states, rewards, hyperparams) fell into place then it became easy for the agent to learn and we had good performance within 20-50 episodes.

The solution itself was pretty simple, all 4 rotors had to be at max speed from the start. But it was seen that not everytime was the agent able to arrive at the optimal solution. Sometimes it will be stuck in a local minima and won't come out of it. Restarting the simulation will solve the issue. So it was kind of 50/50 whether during one run of 200 episodes the agent will arrive at the solution or not.

If it did arrive at the solution then it would learn gradually. It will stay around the optimal solution for a good number of episodes (around 50) then it will diverge and the performance will become very bad. It was seen that the agent was able to come back to the optimal solution if it didn't run out of episodes.

Please see the 2nd plot in "Plot the Rewards" section. It has 2 areas where the performance is good. In the 1st area it was able to find the optimal solution but then it diverges. It is able to reach the optimal solution again and this time it stays there.

The main reason for divergence seemed to be the overestimation of Q values. Please look at the output from "Define the Task, Design the Agent, and Train Your Agent!" section. Whenever the best agent score changed then the corresponding Q value for the reset state of the task was printed. It was seen that Q value will rise exponentially and so will the agent score and the agent will learn quickly. This can be seen b/w episodes 3 and 15.

Then it seemed like Q value reached a peak (it reached around 9x10^7)and after that it started coming down, the agent stayed converged even when the q value was coming down (as can be seen for episode 34 and 44 both of whom have Q value lesser than episode 15 but the score is higher). But after Q value drops enough then agent will lose the convergence and score will drop suddenly. It was seen that then eventually Q value will rise again and this time the Q value will be a lot more stable and agent will stay converged.

Question 4: Briefly summarize your experience working on this project. You can use the following prompts for ideas.

  • What was the hardest part of the project? (e.g. getting started, plotting, specifying the task, etc.)
  • Did you find anything interesting in how the quadcopter or your agent behaved?

Answer:

It was the toughest project of the nanodegree course and I had to spend substantial amount of time in order to make the quadcopter fly properly. The hardest part of the project was first to understand the sample DDPG code that was provided. Also, it was not easy to verify if the Actor and Critic models are working properly or not. So if the agent wasn't behaving properly then localizing the issue wasn't easy-the actor, critic models could be wrong;reward function may be wrong;hyperparameters may be wrong;state info being provided may be wrong. Once I was able to understand the sample DPPG code by going through the DDPG paper that was linked and by reading more material online then I concentrated on reward and state functions. I had to constantly tweak them as sometimes it would feel like I am making them too complicated and sometimes there wasn't much relevant information being passed. In the end the state function that worked was simply passing the z-velocity and not confusing the agent with a lot of other parameters. The reward function also mainly depended on the z-velocity with additional reward for acceleration in the z-direction and penalty for acceleration in any of the angular directions.

One very interesting thing that I noticed was that the agent flew the quadcopter higher than what I had thought. I tried with all the rotor motors at max value and the height of the quadcopter went from 10 to 138 only.

But the agent flew it much higher than that, most of the times achieving the max value of 300. It did this by rotating the quadcopter a bit in the phi direction so it was more like an airplane taking off with it's nose pointing upwards. This was very impressive to me and now I feel like it was something obvious. But if I were to tweak the rotors myself then it is highly unlikely that I will be able to fly the quadcopter as high as the agent did.
